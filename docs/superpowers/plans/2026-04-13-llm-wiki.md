# LLM Wiki 구현 계획

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Karpathy의 LLM Wiki 패턴을 기반으로, Obsidian Web Clipper → Claude Code로 동작하는 완전한 개인 지식 컴파일 시스템 구축

**Architecture:** 3계층 구조 — 불변 원본 소스(sources/) + 컴파일된 위키(wiki/) + 운영 규칙(CLAUDE.md). Python 스크립트나 API 키 없이 Claude Code 자체가 LLM 역할을 수행. Obsidian 볼트는 llm-wiki/ 루트 전체를 가리킴.

**Tech Stack:** Markdown, YAML frontmatter, Obsidian, Obsidian Web Clipper 브라우저 확장, Claude Code CLI

## 프론트매터 형식

모든 위키 페이지는 반드시 아래 YAML 프론트매터로 시작해야 한다:

```yaml
---
title: "페이지 제목"
tags: [도메인, 세부태그1, 세부태그2]
updated: YYYY-MM-DD
sources:
  - https://원본-url-1
  - https://원본-url-2
---
```

## 네이밍 규칙

- 파일명: 소문자 kebab-case
- 고유명사도 소문자 (예: `openai-gpt4.md`, `sam-altman.md`)
- 날짜가 중요한 경우: `YYYY-MM-DD-제목.md`
- 공백 대신 하이픈 사용

### 올바른 예
- `llm-reasoning-costs.md`
- `rust-async-patterns.md`
- `korea-ai-policy.md`

### 잘못된 예
- `LLM Reasoning Costs.md` (대문자, 공백)
- `gpt4Analysis.md` (camelCase)

## 교차 참조

- 같은 위키 내 페이지 링크: `[[페이지명]]`
- 섹션 링크: `[[페이지명#섹션]]`
- 외부 링크: `[링크텍스트](https://url)`

## 페이지 작성 원칙

1. **하나의 페이지 = 하나의 주제** — 여러 주제가 섞이면 분리
2. **사실만 기록** — 의견은 `> 인용 블록`으로 구분
3. **[[WikiLinks]] 적극 활용** — 관련 페이지를 연결해 그래프 구성
4. **출처 명시** — 프론트매터 sources에 원본 URL 항상 포함
5. **업데이트 날짜 갱신** — 내용 변경 시 updated 필드 갱신
```

## Task 3: CLAUDE.md 생성 (핵심 파일)

- Create: `CLAUDE.md`

이 파일이 시스템의 두뇌다. Claude Code가 이 디렉터리에서 열릴 때마다 자동으로 로드된다.

- [ ] **Step 1: CLAUDE.md 생성**

`/Users/user/side-projects/llm-wiki/CLAUDE.md` 파일을 아래 내용으로 생성:

```markdown
# LLM Wiki 운영 지침

이 디렉터리는 Karpathy의 LLM Wiki 패턴으로 운영되는 개인 지식 컴파일 시스템이다.
RAG 대신 점진적으로 축적되는 마크다운 위키를 통해 지식을 합성한다.

## 위키 구조

- **위키 페이지**: `wiki/programming/`, `wiki/ai/`, `wiki/politics/`, `wiki/_concepts/`
- **미처리 소스**: `sources/inbox/` — Web Clipper가 여기에 저장
- **처리완료 소스**: `sources/processed/`
- **색인**: `index.md` — 모든 페이지의 한 줄 요약 목록
- **로그**: `log.md` — 추가 전용 작업 이력
- **규칙**: `SCHEMA.md` — 네이밍, 프론트매터, 분류 규칙

## "inbox 처리해줘" 명령 시

1. `sources/inbox/`의 모든 파일 목록 확인
2. `index.md` 읽기 → 기존 위키 페이지 파악
3. 각 inbox 파일을 순서대로 처리:
   a. 파일 내용 읽기
   b. 관련 기존 위키 페이지 파악 (index.md 기반)
   c. 관련 기존 페이지가 있으면 해당 파일 읽기
   d. 편집 제안 작성:
      - 기존 페이지 업데이트 (새 정보 추가, [[WikiLinks]] 연결) or
      - 새 페이지 생성 (SCHEMA.md 프론트매터 형식 준수)
   e. 변경 내용을 명확히 설명하고 **사용자 승인 대기**
   f. 승인된 경우에만 파일 쓰기
4. 모든 inbox 파일 처리 완료 후:
   - 처리된 파일을 `sources/processed/`로 이동
   - `index.md` 업데이트 (새 페이지 추가 또는 기존 항목 갱신)
   - `log.md`에 항목 추가: `## [YYYY-MM-DD] ingest | {변경 요약}`

## "~에 대해 알려줘" / "~찾아줘" / "~검색해줘" 명령 시

1. `index.md` 읽기 → 관련 키워드와 매칭되는 페이지 목록 파악
2. 관련 페이지들 읽기
3. 위키 내용을 바탕으로 답변 합성 후 출력
4. 답변 마지막에 반드시 물어보기: "이 답변을 위키 페이지로 저장할까요? [y/n]"
   - y: SCHEMA.md 규칙에 따라 새 페이지 생성 후 index.md 업데이트

## "~관련 페이지 찾아줘" 명령 시

`index.md`에서 키워드와 관련된 페이지 목록만 출력.
각 항목: `[[페이지명]]` — 한 줄 요약

## "~업데이트됐나?" 명령 시

해당 주제의 위키 페이지 읽기 → `updated` 날짜와 내용 요약 출력.
소스 URL이 있으면 함께 표시.

## "위키 점검해줘" 명령 시

전체 위키 페이지 읽기 후 아래 항목 분석 및 리포트 출력:

1. **모순**: 서로 다른 페이지에서 상충되는 정보
2. **고아 페이지**: index.md에 없거나 아무 페이지도 링크하지 않는 페이지
3. **빈 지식 영역**: index.md에 항목이 없는 도메인 폴더
4. **오래된 페이지**: updated 날짜가 6개월 이상 된 페이지 목록
5. **깨진 링크**: 존재하지 않는 페이지를 참조하는 [[WikiLinks]]

## 위키 페이지 작성 규칙

- 파일명: 소문자 kebab-case (SCHEMA.md 참조)
- 반드시 YAML 프론트매터 포함: title, tags, updated, sources
- Obsidian `[[WikiLinks]]` 적극 활용
- 사실만 본문에 기록, 의견/추측은 `> 인용 블록`으로 구분
- 각 페이지는 단일 주제에 집중
- 변경 시 항상 updated 날짜 갱신

## 중요 원칙

- **사용자 승인 없이 절대 파일 쓰지 않는다** (ingest 포함)
- sources/inbox/ 파일은 읽기만 한다 — 처리 전까지 수정하지 않는다
- sources/processed/ 이동은 모든 관련 위키 편집 완료 후에만 한다
```
